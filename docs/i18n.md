## Overview

planet-webapp uses [next-intl](https://next-intl.dev) for translations and locale-aware rendering. The app runs on the **Pages Router**, which shapes how next-intl is configured compared to App Router projects.

Routing is handled by a **custom middleware** (`middleware.ts`) rather than next-intl's built-in routing. The middleware resolves the tenant from the hostname, negotiates the locale, and rewrites every request to an internal multi-tenant path (`/sites/[slug]/[locale]/...`). next-intl's middleware has no concept of a tenant slug, so it cannot be used here.

## Key files

| File | Purpose |
|---|---|
| `i18n.ts` | Server-side runtime config — loads and merges message JSON files for each request |
| `i18n-config.ts` | Locale list and default locale used across the app |
| `src/i18n/types.ts` | Compile-time type augmentation — gives `useTranslations` key-level autocomplete |
| `middleware.ts` | Custom locale detection and routing (does not use next-intl middleware) |
| `src/utils/language/getMessagesForPage.ts` | Helper used in `getStaticProps` to load only the message files a page needs |
| `src/utils/getLocalizedPath.ts` | Builds a locale-prefixed path; strips any existing locale segment first |
| `src/hooks/useLocalizedPath.ts` | React hook wrapping `getLocalizedPath` with the current locale from `useLocale()` |

## Request flow

There are two separate concerns: **routing** (middleware, runs on every request) and **message loading** (getStaticProps, runs at build time or on the server per page).

### 1. Routing — middleware

Take a request for `/fr/donana-restoration-project` on `trilliontreecampaign.org`:

1. Middleware resolves the tenant slug from the hostname → `ttc`.
2. If the URL has no locale prefix, middleware picks one from the `NEXT_LOCALE` cookie or `Accept-Language` header and **redirects** (e.g. `/donana-restoration-project` → `/fr/donana-restoration-project`).
3. Once a locale is present, middleware **rewrites** the URL internally: `/fr/donana-restoration-project` → `/sites/ttc/fr/donana-restoration-project`. The user never sees the rewritten path.
4. Next.js matches the rewritten path to `pages/sites/[slug]/[locale]/[p].tsx` with `slug=ttc`, `locale=fr`, `p=donana-restoration-project`.

### 2. Message loading — getStaticProps

Each page's `getStaticProps` calls `getMessagesForPage` with the locale and the specific message files that page needs:

```ts
const messages = await getMessagesForPage({
  locale: 'fr',
  filenames: ['common', 'projectDetails'],
});
```

`getMessagesForPage` loads both the locale files (`fr/`) and the English defaults (`en/`), then merges them with `deepmerge`. English fills in any keys missing from the locale files. The result is returned as `pageProps.messages`.

### 3. Rendering

`_app.tsx` wraps the app in `NextIntlClientProvider`, passing the locale from `router.query.locale` and the messages from `pageProps.messages`. Components then call `useTranslations('Namespace')` to read strings and `useLocale()` to read the current locale.

### Where i18n.ts fits in

`i18n.ts` is next-intl's server-side config (`getRequestConfig`). In Pages Router its role is narrow: it provides locale context for SSR so that server-side hooks like `useLocale()` work during the initial render. It is **not** the source of messages for components — that always flows through `getStaticProps` → `pageProps.messages` → `NextIntlClientProvider`.

### Why not next-intl middleware or routing?

All pages live under `pages/sites/[slug]/[locale]/` — the `[slug]` segment is the tenant identifier resolved from the hostname. next-intl's routing layer has no concept of a tenant slug: it expects locale-prefixed paths in the form `/[locale]/...` and cannot inject an extra segment.

The custom middleware (`middleware.ts`) handles all three responsibilities in one pass:

1. **Tenant resolution** — looks up the tenant slug and its supported locales from the hostname.
2. **Locale negotiation** — selects the best locale from the URL path, `NEXT_LOCALE` cookie, or `Accept-Language` header using `@formatjs/intl-localematcher` and `Negotiator`.
3. **Rewrite** — rewrites the public URL (`/[locale]/path`) to the internal path (`/sites/[slug]/[locale]/path`) that Next.js file routing expects.

## Supported locales

Defined in `i18n-config.ts`:

| Code | Language |
|---|---|
| `en` | English (default) |
| `de` | German |
| `cs` | Czech |
| `es` | Spanish |
| `fr` | French |
| `it` | Italian |
| `pt-BR` | Portuguese (Brazil) |

Each tenant may support a subset of these. The middleware filters `i18nConfig.locales` to only the languages the tenant supports before negotiating the locale.

## Message files

Locale files live in `public/static/locales/<locale>/`. Each file covers a feature area (e.g. `common.json`, `projectDetails.json`). Pages load only the files they need — see the `filenames` array in each page's `getStaticProps`.

The English files (`public/static/locales/en/`) are the source of truth for all keys. Other locales are partial — missing keys fall back to English via the `deepmerge` in `getMessagesForPage`.

## Type safety

`src/i18n/types.ts` uses `typeof import(...)` to derive a type from each English JSON file, intersects them into a single `Messages` type, and registers it with next-intl:

```ts
declare module 'next-intl' {
  interface AppConfig {
    Messages: Messages;
  }
}
```

This makes `useTranslations('ProjectDetails')` aware of the exact keys in `projectDetails.json` — misspelled keys are a TypeScript error. The `export {}` at the bottom of the file is required to make TypeScript treat it as a module (not a global script), so the `declare module` block augments next-intl rather than overriding it.

## Usage

### Hooks and utilities

#### `useTranslations`

```ts
import { useTranslations } from 'next-intl';

const t = useTranslations('Common');
t('save'); // key from common.json → { "Common": { "save": "Save" } }
```

Use a separate `t` per namespace if a component spans multiple files:

```ts
const tCommon = useTranslations('Common');
const tDonate = useTranslations('Donate');
```

JSON files can have nested objects. Pass a dot-separated path to scope to a nested namespace:

```ts
// Scoped to Profile.myContributions — keys are just the leaf names
const t = useTranslations('Profile.myContributions');
t('treesContributed', { count: 5 }); // → "5 trees"
```

#### `useLocale`

```ts
import { useLocale } from 'next-intl';

const locale = useLocale(); // → "fr"
```

#### `useLocalizedPath` / `getLocalizedPath`

Use the hook inside components to prepend the current locale to a path:

```ts
import useLocalizedPath from '../hooks/useLocalizedPath';

const { localizedPath } = useLocalizedPath();
localizedPath('/profile'); // → "/fr/profile"
```

Use `getLocalizedPath` outside React (e.g. in `getStaticProps` or redirect logic):

```ts
import { getLocalizedPath } from '../utils/getLocalizedPath';

getLocalizedPath('/profile', 'de'); // → "/de/profile"
```

### ICU message patterns

Messages use the [ICU message format](https://unicode-org.github.io/icu/userguide/format_parse/messages/). The following patterns are used across the codebase.

#### Interpolation

Pass values for `{placeholder}` variables as the second argument:

```json
"hoursAgo": "{age}h ago"
```

```ts
t('hoursAgo', { age: '3' }); // → "3h ago"
```

#### Number formatting

`{value, number}` formats a number per the active locale's conventions:

```json
"restoredAreaAgainstTarget": "{count, number} of {unit, number} m² restored"
```

```ts
t('restoredAreaAgainstTarget', { count: 1000, unit: 2500 });
// en → "1,000 of 2,500 m² restored"
// de → "1.000 of 2.500 m² restored"
```

#### Plural

`#` is replaced by the count value. Use `=1` for an exact match or CLDR keywords (`one`, `other`, `few`, etc.) for locale-aware pluralisation:

```json
"filterResult": "{count, plural, =1 {# Project} other {# Projects}} found"
```

```ts
t('filterResult', { count: 1 }); // → "1 Project found"
t('filterResult', { count: 5 }); // → "5 Projects found"
```

#### Select

Branch on a non-numeric value. `other` is the required fallback:

```json
"name": "{type, select, individual {Name} organization {Company Name} other {Name}}"
```

```ts
t('name', { type: 'organization' }); // → "Company Name"
```

#### Rich text

Use `t.rich()` when a message contains inline markup. Provide a render function for each tag name in the JSON value — the tag name is arbitrary, it just has to match:

```json
"highAlertConfidenceText": "<important>High</important> alert confidence"
```

```tsx
t.rich('highAlertConfidenceText', {
  important: (chunks) => <span className={styles.bold}>{chunks}</span>,
});
// → <><span className="bold">High</span> alert confidence</>
```

#### Combining patterns

ICU patterns can be nested, and rich text tags work alongside them:

```json
"totalPlantedTrees": "{count, plural, =1 {# Tree} other {{formattedCount} Trees}} {area, select, undefined {} other {<areaContainer>({area} ha)</areaContainer>}}"
```

```tsx
t.rich('totalPlantedTrees', {
  count: 3,
  formattedCount: '3',
  area: '1.5',
  areaContainer: (chunks) => <span className={styles.area}>{chunks}</span>,
});
// → <>3 Trees <span className="area">(1.5 ha)</span></>
```

### Maintenance

#### Adding a translation key

1. Add the key to `public/static/locales/en/<file>.json` (English is the source of truth).
2. Add the translated string to the equivalent file in each other locale folder. If omitted, the English value is used as fallback.
3. TypeScript picks up the new key automatically — no changes to `src/i18n/types.ts` needed.

#### Loading a message file in a new page

Pass filenames (without `.json`) to `getMessagesForPage` in the page's `getStaticProps`. Only the files listed here are loaded for that page.

```ts
const messages = await getMessagesForPage({
  locale: context.params?.locale as string,
  filenames: ['common', 'projectDetails'],
});
```

Available filenames are enforced by the `TRANSLATION_FILE_NAMES` union type in `src/utils/language/getMessagesForPage.ts`:

`allProjects` · `bulkCodes` · `common` · `country` · `donate` · `donationLink` · `donationReceipt` · `editProfile` · `giftfunds` · `leaderboard` · `managePayouts` · `manageProjects` · `maps` · `me` · `planet` · `planetcash` · `profile` · `projectDetails` · `project` · `redeem` · `tenants` · `treemapper`

#### Adding a new locale file

1. Create `public/static/locales/en/<newFile>.json` with your keys.
2. Add the corresponding file in each locale folder.
3. Add the filename to the `TRANSLATION_FILE_NAMES` union in `src/utils/language/getMessagesForPage.ts`.
4. Register it in `i18n.ts` — add the import to both `userMessages` and `defaultMessages`.
5. Add a `type MessagesNewFile = typeof import('...<newFile>.json')` entry and include it in the `Messages` intersection in `src/i18n/types.ts`.

## Troubleshooting

### Translations show raw keys

**Symptom:** A component renders the key string (e.g. `"Common.save"`) instead of the translated text. No error is thrown.

**Cause:** The message file that contains the key was not included in the page's `getStaticProps`.

**Fix:** Add the filename to the `filenames` array:

```ts
const messages = await getMessagesForPage({
  locale: context.params?.locale as string,
  filenames: ['common', 'projectDetails'], // add the missing file here
});
```

Check the `TRANSLATION_FILE_NAMES` union in [src/utils/language/getMessagesForPage.ts](../src/utils/language/getMessagesForPage.ts) for the list of valid names.

---

### Runtime error on a translated string

**Symptom:** next-intl throws at render time, typically with a message like `"The intl string context variable 'count' was not provided"` or an ICU parse error.

**Common causes:**

| Cause | Example |
|---|---|
| Required placeholder missing | `t('filterResult')` — forgot `{ count }` |
| Placeholder name mismatch | JSON has `{count}` but code passes `{ total }` |
| `t.rich()` used for a key that has no tags | No tag render functions needed; use `t()` instead |
| Malformed ICU in the JSON file | Unclosed `{`, mismatched braces |

**Fix:** Compare the JSON key's ICU pattern against the call site. Every `{placeholder}` in the JSON must have a matching key in the second argument to `t()` or `t.rich()`.

---

### Key autocomplete not working for a new file

**Symptom:** `useTranslations('NewNamespace')` does not offer key autocomplete in the IDE, or TypeScript does not catch misspelled keys.

**Cause:** The new JSON file was not registered in `src/i18n/types.ts`.

**Fix:** Add the type alias and include it in the `Messages` intersection — see step 5 of [Adding a new locale file](#adding-a-new-locale-file).

---

### User is always redirected to English

**Symptom:** A user with a non-English browser language is redirected to `/en/...` instead of their preferred locale.

**Cause:** The middleware negotiates a locale from the `NEXT_LOCALE` cookie, then the `Accept-Language` header, filtered against the tenant's supported locales. If none match, it falls back to `en`.

**Fix:** Check that the expected locale is in `i18nConfig.locales` in `i18n-config.ts` and that the tenant's config includes it as a supported locale. Inspect the `NEXT_LOCALE` cookie and the `Accept-Language` request header in the browser dev tools to see what the middleware is working with.
